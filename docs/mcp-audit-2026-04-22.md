## Findings

### Criterion 1 — Tool Abstraction [9/10]
Tools are framed around agent intent ("browse", "add", "search", "call") rather than REST verbs/paths. There is no 1:1 mapping to Supabase tables or HTTP endpoints — the gateway correctly hides the catalogue / users / upstream-dispatch layers behind task-level actions. Minor deduction: `call_tool` is a thin passthrough that exposes the upstream's tool namespace to the model, which leaks the underlying transport model to some degree.

### Criterion 2 — Composability & Structured Output [3/10]
No `outputSchema` is declared on any tool. Every result is serialised with `json.dumps(...)` into a plain `TextContent` string (`src/gateway/routes.py:231-297`). The model must re-parse JSON on every hop — `browse_mcps` → `add_mcp` → `list_tools` → `call_tool` is a four-step chain where each step discards type information. There is no REPL or code-execution affordance that would let the agent script the whole flow in one call.

### Criterion 3 — Tool Count & Context Hygiene [9/10]
Exactly 7 tools, all essential — well within the ≤10 sweet spot. No deferred-loading annotations are needed at this size. The `search_tools` tool effectively acts as progressive disclosure for the (potentially large) upstream tool surface, which is the right pattern.

### Criterion 4 — Description Quality [4/10]
Descriptions are one-line stubs. None carry examples, return-shape hints, failure modes, or "when to use" guidance. Parameters have no `description` fields at all — e.g. `call_tool` declares `{"mcp_slug": {"type": "string"}, "tool_name": {"type": "string"}, "arguments": {"type": "object"}}` with zero context. Best of the lot: `add_mcp` at least cross-references `browse_mcps`. Worst: `call_tool` — "Call a tool on an upstream MCP server." — gives the model no clue how to choose `mcp_slug`/`tool_name` or shape `arguments`.

### Criterion 5 — Agent-Oriented Design [5/10]
Server name `"DS-MOZ Intelligence Gateway"` is set (`src/gateway/routes.py:147`) but no `instructions=` / `setInstructions` is passed. The agent is dropped into the tool list with no orientation — it must infer the browse → add → search → call workflow from tool names alone. Tool descriptions describe *what* not *when*. No hints that meta-tools are free while `call_tool` is credit-gated, which is a material fact the agent should know before looping.

### Criterion 6 — Overlap & Ambiguity Risk [5/10]
Internally coherent, but the tool name `list_tools` collides with MCP's own protocol-level `tools/list` method and with almost every other server that wraps an API. `call_tool` has the same problem — it shadows MCP's `tools/call`. In a mixed client setup (this gateway + another MCP), the agent will have no way to disambiguate. `list_mcps` vs `browse_mcps` is a subtler duplication: the difference ("in your toolbox" vs "all available") needs to be inferred from two short sentences.

## Priority Recommendations

1. **Rewrite every tool description as a prompt.** *(Criterion 4 — +4 pts)*
   Each description should carry: one-sentence purpose, when to use it vs a sibling tool, a worked example of inputs/outputs, and failure modes. Add `description` to every parameter in `inputSchema` (`src/gateway/routes.py:152-210`). For `call_tool`, explicitly say "this is credit-gated; meta-tools are free" and "use `list_tools` first to discover valid `tool_name` and argument shape".

2. **Rename `list_tools` and `call_tool` to avoid protocol collision.** *(Criterion 6 — +3 pts)*
   Suggested: `list_mcp_tools` (or `describe_mcp_tools`) and `invoke_mcp_tool`. The current names shadow `tools/list` and `tools/call` from the MCP spec itself, which will confuse both models and human readers of logs.

3. **Add server-level `instructions` covering the workflow.** *(Criterion 5 — +3 pts)*
   Pass `instructions=` when constructing `Server(...)` at `src/gateway/routes.py:147`. Cover: the browse→add→search→call lifecycle, credit semantics, the fact that `list_mcps` shows *enabled* MCPs while `browse_mcps` shows *all*, and that the toolbox persists across sessions.

4. **Declare `outputSchema` on every tool.** *(Criterion 2 — +3 pts)*
   Define typed returns (e.g. `{mcp, name, description, category, enabled}` for `browse_mcps`) so downstream calls can chain without re-parsing JSON. The MCP Python SDK supports structured content; returning a plain JSON string inside `TextContent` is legacy.

5. **Surface credit cost in `browse_mcps` / `list_mcps`.** *(Criterion 4/5 — +1 pt)*
   The agent currently cannot see `credit_cost_per_call` until `call_tool` fails with "Insufficient credits". Include the cost per call in the listing payload so the model can plan.
